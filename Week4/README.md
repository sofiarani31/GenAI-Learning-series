# Week 4: Chat with Your PDF Knowledge Base

Welcome to the **Retrieval-Augmented Generation (RAG)** project! This application is a web-based chat interface built with **Streamlit** that allows users to upload PDF documents, index their contents, and ask questions about them. The app uses an LLM (Google Gemini) combined with a vector database (Pinecone) to generate answers grounded purely in the provided documents.

---

## 🚀 Features

- **Multiple PDF Uploads:** Support for ingesting multiple PDF documents at once.
- **Robust Text Extraction:** Handles text extraction gracefully and skips corrupt, empty, or unreadable PDFs without crashing.
- **Semantic Search:** Uses `sentence-transformers` (`all-MiniLM-L6-v2`) to turn text chunks into embeddings for highly accurate semantic retrieval.
- **Vector Storage:** Stores all embeddings and metadata (page numbers, chunk IDs) in **Pinecone Serverless**.
- **Source Citation:** Every answer generated by the LLM includes an expandable "View Sources" section linking back to the exact page and text snippet used to formulate the answer.
- **Conversation Memory:** Keeps track of chat history, meaning you can ask follow-up questions seamlessly.
- **Dynamic Re-indexing:** If you upload a newer version of a previously uploaded PDF, the app automatically deletes the old vectors before indexing the new one to prevent duplication.

## 🧠 What I Learned This Week

Building this project provided hands-on experience with the entire RAG pipeline from scratch:

1. **Document Ingestion & Chunking:**
   - How to extract text while maintaining crucial metadata (like page numbers) for future citations.
   - Why we split text into overlapping chunks (e.g., 500 characters with 100 character overlap) so sentences aren't randomly cut in half.

2. **Embeddings & Vector Databases:**
   - Why the **same** embedding model must be used for both indexing chunks and encoding user queries.
   - How to configure and upsert data into a Pinecone Index.
   - How cosine similarity is used to find the most mathematically "close" chunks of text to a user's question.

3. **Prompt Engineering for Grounding:**
   - How to inject retrieved context directly into an LLM prompt.
   - Using strict XML-like prompt boundaries (e.g., `<Context>`) to prevent hallucinations and force the LLM to admit when it *doesn't* know the answer.

4. **Streamlit UI/UX Patterns:**
   - Managing application state (`st.session_state`) so chat history isn't wiped on every button click.
   - Cleaning up temporary files (like wiping the `uploads/` folder before saving new ones).
